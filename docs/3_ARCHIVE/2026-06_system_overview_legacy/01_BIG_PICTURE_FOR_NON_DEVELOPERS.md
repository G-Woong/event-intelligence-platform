# 비개발자를 위한 시스템 전체 그림

> 이 문서는 프로그래밍 지식 없이도 시스템의 목적과 구조를 이해할 수 있도록 작성되었습니다.

---

## 이 시스템은 무엇을 합니까?

**전세계에서 매일 수천 건씩 발생하는 뉴스·공시·이벤트를 자동으로 수집하고, AI가 분석해서 사람이 빠르게 읽을 수 있는 "사건 카드" 형태로 가공해 웹 화면에 보여주는 인텔리전스 서비스입니다.**

> **중요**: 이 시스템은 **정보 제공**이 목적입니다. 투자 권유·매수/매도 추천·금융 조언을 제공하지 않습니다. 시장 움직임을 언급할 때도 "좋다/나쁘다"와 같은 가치 판단을 덧붙이지 않습니다.

---

## 뉴스룸 비유로 이해하기

이 시스템을 **자동화된 글로벌 뉴스룸**에 비유할 수 있습니다.

```
[기자들]         [편집부]           [데스크]            [독자]
  |                 |                  |                  |
RSS 피드    →   수집 로봇   →    AI 분석팀   →    웹 화면
(BBC, Reuters,  (worker)       (agent-worker)   (Next.js)
 연합뉴스 등)                    LangGraph AI
```

| 뉴스룸 역할 | 이 시스템에서의 역할 |
|---|---|
| 외신 전신을 받아오는 텔레타이프 | RSS 피드 수집기 (rss_collector) |
| 중복 기사 걸러내는 편집부 | 중복 제거 로직 (content_hash) |
| 사실 확인하는 팩트체커 | LangGraph fact_check 노드 |
| 카테고리 분류하는 에디터 | LangGraph sector_mapping 노드 |
| 영향 분석하는 전문기자 | LangGraph impact_analysis 노드 |
| 최종 기사 쓰는 기자 | LangGraph final_card_writer 노드 |
| 독자가 읽는 신문/웹사이트 | Next.js 프론트엔드 |
| 편집장 대시보드 | Admin 페이지 |

   ┌─────────────────────────────────────────┐

                          ▼

   ┌─────────────────────────────────────────┐
   │  1. 출처 파싱 → 2. 정규화 → 3. 중복제거  │

   │  10. 최종 카드 작성 → 11. 발행 결정      │
   └──────────────────────┬──────────────────┘

   ┌─────────────────────────────────────────┐
   │  • event_cards 테이블 (PostgreSQL)       │
   │  • Milvus — "의미 유사도 기반 검색"      │
   │  • OpenSearch — "키워드 기반 검색"       │
   └──────────────────────┬──────────────────┘
                          │
                          ▼
⑥ API 제공
   ┌─────────────────────────────────────────┐

## 사용자가 보게 되는 5가지 화면

### 1. 홈 화면 (`/`)
- 최근 수집·분석된 사건 카드 목록
- 각 카드: 제목, 출처, 테마, 영향 요약

### 2. 사건 목록 (`/events`)
- 전체 이벤트 목록
- 테마별·섹터별 필터, 키워드 검색 지원

### 3. 사건 상세 (`/events/[id]`)
- 개별 사건 카드 전체 내용
- 출처 링크, AI가 분석한 영향, 관련 엔티티

### 4. 테마/섹터 페이지 (`/themes`, `/sectors`)
- 지정학·거시경제 등 테마별 이벤트 묶음
- 에너지·금융·기술 등 섹터별 이벤트 묶음

### 5. 검색 페이지 (`/search`)
- 키워드로 전체 사건 카드 검색
- OpenSearch 기반 빠른 텍스트 검색

### (관리자 전용) Admin 페이지 (`/admin`)
- 수집된 원시 데이터 확인
- 색인 재구축, 처리 재시도, 대기열 상태 조회

## 지금 시스템의 한계 (솔직한 현황)

현재 구현은 **스켈레톤(뼈대) 단계**입니다. 아래 기능은 아직 임시(mock) 상태입니다:

| 기능 | 현재 상태 | 의미 |
|---|---|---|
| AI 엔티티 추출 | 임시 | 고유명사 추출이 가짜 값 반환 |
| 카테고리 분류 | 임시 | 키워드 매칭 수준 |
| 영향 분석 | 임시 | 실제 LLM 대신 템플릿 반환 |
| 팩트체크 | 임시 | 외부 검증 없이 항상 통과 |
| 최종 카드 작성 | 임시 | AI 요약 없이 원문 그대로 |
| 수집 소스 | 부분 | RSS 3개 소스만 (DART·SEC 미구현) |

→ 이 한계들이 해소되는 로드맵: **[11_NEXT_ENHANCEMENT_ROADMAP.md](./11_NEXT_ENHANCEMENT_ROADMAP.md)**

## 시스템의 강점 (이미 동작하는 것들)

- **10개 컨테이너**가 동시에 건강하게 실행 중 (모든 healthcheck PASS)
- **13단계 파이프라인**이 처음부터 끝까지 연결됨 (RSS 입력 → 화면 출력)
- **PostgreSQL + Milvus + OpenSearch** 3중 저장소 모두 연결됨
- **중복 제거** 로직으로 같은 기사가 반복 저장되지 않음
- **Admin 운영 도구** (재시도·재색인·상태 조회) 모두 동작
- **보안**: 외부 DB 포트는 모두 서버 내부로만 노출, API 키는 환경변수로만 관리
