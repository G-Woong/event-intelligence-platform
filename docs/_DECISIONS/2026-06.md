### [#1] 턴 종료 하네스를 "훅(사실) + 스킬(의미) + fleet(감사)" 3계층으로 구축
- 날짜: 2026-06-19 · 섹터: harness · 상태: 완료

**문제** — 매 턴 끝에 ①진행리포트 ②docs 동기화 ③팀 감사 ④risk 수집을 "누락 없이·파일 증식 없이·비개발자도 읽게" 자동화해야 하는데 단일 메커니즘으론 불가.
**배경** — 사용자가 "프로젝트 주도권"을 유지하려면 매 턴 신뢰할 현황 + 세션이 바뀌어도 복원되는 사고 흐름이 필요. 프로젝트 전체 운영 기반(하네스 자산).
**가설** — 하네스 제약 2개(훅은 서브에이전트 호출 불가 / 훅은 의미 서술 불가)가 단일 메커니즘을 막는 원인이라 추정.
**도입 아이디어** — 계층 분리: 훅=사실(결정론·신뢰), 스킬=의미(LLM 서술), fleet=조건부 감사.
**선택 이유** — 대안 (A)순수 훅=의미 서술 불가, (B)순수 에이전트 규율=확률적 누락, (C)Stop block 강제=루프 위험. 제약상 (D)계층 분리만이 사실 신뢰성과 의미 해석성을 동시 충족.
**구현** — `turn_state_snapshot.py`(훅) + `turn-closeout`(스킬) + 기존 15 에이전트 라우팅.
**결과** — 훅 1·스킬 1·config 1 최소 표면. 사실 자동(`machine_status.json`), 의미 분리(`PROJECT_STATUS`/`_DECISIONS`/`_RISK`).
**한계** — 의미 계층은 best-effort(에이전트가 closeout 실행해야). soft nudge로 환기.
**후속 과제** — 라이브 다턴 관찰.
**관련 문서** — `docs/Harness_Construction/00,05`.

### [#2] 팀 다각도 감사로 설계 검증 후 10개 risk 반영(rev2) — 단일파일 race 해소
- 날짜: 2026-06-19 · 섹터: harness · 상태: 완료

**문제** — 초안(rev1)은 훅·에이전트가 같은 `PROJECT_STATUS.md`를 동시 write(race), 트리거가 `git diff HEAD`라 untracked/한글경로 누락.
**배경** — 사실 계층 신뢰성이 무너지면 하네스 전제가 붕괴. 비개발자에게 거짓 안심 유발.
**가설** — 적대/구조/보안 3에이전트 감사로 "동작하지 않는 지점"을 조기 발견 가능하다고 추정.
**도입 아이디어** — writer 단일화(파일 분리): 훅=`machine_status.json`, 에이전트=`PROJECT_STATUS.md`+`narrative_marker.json`. 경로파싱 `porcelain -z`+UTF-8. turn=session별 map.
**선택 이유** — 대안(단일파일 락) 대신 파일 분리 — 락보다 단순하고 race 원천 제거. mtime freshness는 자가당착이라 폐기.
**구현** — 훅/스킬/설계도 rev2. 보안 게이트 APPROVED.
**결과** — 치명 3건 수정·단위검증 통과(severity 오탐0, marker 분리 fresh 전이, turn map).
**한계** — 단일 writer는 규율로 보장(경로 게이트 없음). audit_required 하네스 자기수정 중 고착(soft라 수용).
**후속 과제** — 활성화 후 라이브 관찰.
**관련 문서** — `docs/Harness_Construction/01,03,05`.

### [#4] nudge를 "정적 dirty tree"가 아니라 "마지막 마감 이후 새 작업(sig)"에 반응하도록 변경(rev3)
- 날짜: 2026-06-19 · 섹터: harness · 상태: 완료

**문제** — 활성화 후 turn-closeout nudge가 매 턴 반복 발화.
**배경** — 무의미한 매 턴 알림은 신호를 소음화 → "사소한 작업은 판단" 원칙·UX 위반. 하네스 신뢰성 문제.
**가설** — nudge 조건이 `전체 미커밋 변경(상존)`에 반응 + freshness가 1턴만 유지 → 새 작업 없어도 매 턴 발화로 추정(적대 검토 M2).
**도입 아이디어** — 변경 signature(sig) 기반: 서술 산출물·gitignored 제외한 비-서술 파일 집합이 마지막 마감 이후 **바뀔 때만** nudge.
**선택 이유** — 대안 (A)커밋(사용자 승인 필요) (B)턴카운터 유지(반복 불가피). (C)sig가 "새 작업"의 본질에 부합.
**구현** — 훅에 `_is_narration_output` 제외 + sig 비교 freshness. 스킬은 `narrated_sig` 기록.
**결과** — marker에 sig 기록 후 `narrative_fresh=True`·nudge 소멸 검증. 정적 dirty tree 침묵.
**한계** — 기존 untracked 파일의 내용-only 변경(경로 불변)은 미포착(과소 nudge).
**후속 과제** — 내용-only 변경 과소 nudge 운영 관찰.
**관련 문서** — `docs/Harness_Construction/01,05`.

### [#5] (근본원인) Stop 훅 `stop_hook_active` 미검사 → 턴 종료 무한 차단(9회 강제종료)
- 날짜: 2026-06-19 · 섹터: harness · 상태: 완료

**문제** — 하네스 활성화 후 한 턴이 끝나지 못하고 9회 재호출된 뒤 강제 종료("A hook blocked the turn from ending 9 consecutive times"). ~31분 churn.
**배경** — 사용자가 새 지시를 못 내리고 세션이 마비 = 하네스가 작업을 차단. 요구 달성의 직접 방해.
**가설** — 처음엔 "dirty tree 알림 소음"으로 오판. 하네스 강제종료 메시지가 진짜 원인 지목: **Stop 훅이 출력(additionalContext)을 내면 턴 종료가 차단되고 재호출**되는데, 세 훅이 `stop_hook_active` 미검사라 continuation 중에도 또 출력 → 무한 차단.
**도입 아이디어** — 하네스가 직접 지시한 공식 패턴: 모든 Stop 훅이 `stop_hook_active==true`면 출력 없이 즉시 return.
**선택 이유** — 대안(additionalContext 전면 제거=알림 상실)·(config로 끄기=가치 상실) 대비, 가드만이 알림 유지 + 루프 차단.
**구현** — 3개 Stop 훅(secret_scan_reminder·docs_conflict_grep_check·turn_state_snapshot) 전부 payload 파싱 직후 `if stop_hook_active: return 0`. turn_state는 스냅샷 기록은 유지, nudge stdout만 가드.
**결과** — 검증: stop_hook_active=true 시 3개 훅 무출력(턴 종료 허용). 턴당 최대 1회 알림 후 종료 → 9회 루프 불가.
**한계** — dirty tree면 턴당 1회 continuation 잔존(알림 전달용). 완전 0은 커밋으로 트리 청소 시.
**후속 과제** — 향후 Stop 훅 추가 시 이 가드 기본 포함.
**관련 문서** — `docs/Harness_Construction/05`(필수 가드 명문화).

### [#6] 하네스를 shell-hook 자동화 → stamp-gated agent-orchestrated closeout으로 격상(Option C)
- 날짜: 2026-06-19 · 섹터: harness · 상태: 완료

**문제** — 1차 하네스는 사실 수집은 자동이나 의미해석·팀감사·code-review가 main agent의 자율 실행에 의존(강제 X), code-review 스킬 미연결, 완료 증거(stamp) 부재. "매 턴 무조건"이 reminder 수준이었다.
**배경** — 웹 인텔리전스 프로젝트의 실제 운영 백본이 되려면 closeout이 "실제로 끝났는가"를 기계적으로 검증해야 한다. shell hook 단독 사고로는 의미판단/감사를 못 한다.
**가설** — hook=센서/게이트, main agent=오케스트레이터, skills=절차, subagents=감사단, stamp=완료증거로 역할을 분리하면 자동성과 판단력을 동시에 얻는다.
**도입 아이디어** — Option C(stamp-gated hybrid): 훅이 `audit_required.json`(PostToolUse)·`machine_status.audit_types`(Stop)에 flag만 세우고, `turn-closeout`이 flag→실제 에이전트/`/code-review` 라우팅 호출 후 `closeout_stamp.json`에 완료증거 기록. Stop hook은 stamp.working_tree_signature vs 현재 sig mismatch를 게이트로 감지.
**선택 이유** — A(hook-heavy)=의미판단 불가+파괴위험, B(agent-orchestrated)=강제장치 부재. C만이 자동감지+agent판단+완료증거를 결합. mismatch가 stamp에 남아 "감사 누락"이 자동 적발됨.
**구현** — Stop hook에 `audit_types`+stamp-gate, 신규 PostToolUse `audit_flagger.py`, `turn-closeout` 오케스트레이터 확장, `03` flag→실제15에이전트 매핑, `closeout_stamp.json` 스키마, dead-code 스캐너(`scripts/dead_code_scan.py`) 배선, _DECISIONS 월별 ledger 전환.
**결과** — flag→라우팅→stamp 게이트 체인 구성. 미완 closeout이 다음 턴 mismatch로 자동 재적발되는 구조.
**한계** — stamp 기록·subagent 호출의 실행 주체는 여전히 main agent(soft). 단 미실행이 stamp/sig mismatch로 "증거에 남아" 은폐 불가 — 1차의 "조용한 누락"보다 강함.
**후속 과제** — dead-code AST 도구(vulture) 배선, code-review 스킬 라이브 호출 관찰.
**관련 문서** — `docs/Harness_Construction/03·05`, `.claude/skills/turn-closeout/SKILL.md`.
