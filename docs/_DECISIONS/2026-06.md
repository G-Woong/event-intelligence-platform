**구현** — 훅에 `_is_narration_output` 제외 + sig 비교 freshness. 스킬은 `narrated_sig` 기록.
**결과** — marker에 sig 기록 후 `narrative_fresh=True`·nudge 소멸 검증. 정적 dirty tree 침묵.
**한계** — 기존 untracked 파일의 내용-only 변경(경로 불변)은 미포착(과소 nudge).
**후속 과제** — 내용-only 변경 과소 nudge 운영 관찰.

### [#7] closeout 게이트를 "경로 sig + 자기보고" → "content-hash sig + 구조화 evidence"로 격상(증거기반)
- 날짜: 2026-06-19 · 섹터: harness · 상태: 완료

**문제** — 직전 stamp 게이트는 (a) sig가 경로집합 기반이라 **동일 경로의 내용-only 변경**(claim/risk 근거/config)을 못 잡고, (b) `audit_types_addressed`에 유형명만 적으면 통과해 **감사를 실제로 안 해도 거짓 stamp가 통과**했다.
**배경** — 운영 백본이 "증거 기반 stamp-gated"가 되려면, 무엇이 바뀌었는지(내용)와 무엇을 검토했는지(증거)를 게이트가 기계적으로 확인해야 한다. 직전 감사가 R-CloseoutTrust로 이 둘을 잔여 gap으로 남겼었다.
**가설** — sig에 중요 파일 content-hash를 합성하면 내용변경이 잡히고, required 감사마다 구조화 evidence(executed+verdict)를 요구하면 자기보고 위조 비용이 오른다. 단 "LLM이 실제로 추론했는가"는 원천적으로 검증 불가.
**도입 아이디어** — ① `compute_signature`(경로 sig + `content:<path>#hash` of 중요 파일군, narration 포함) + 에이전트가 편집 후 `scripts/closeout_sig.py`로 재계산해 stamp에 복사(hook과 동일 함수→수렴). ② `_audit_attested` 게이트(required audit별 evidence 필수). ③ `scripts/harness_doctor.py`로 gitignored settings.json 재현성 점검. ④ dead_code_scan에 ruff 연동.
**선택 이유** — 대안 A(전체 repo hash)=과중·narration 루프, 대안 B(별도 감사 리포트 파일 요구)=`03 §4`(파일 증식 금지)와 상충. 중요 파일군만 hash + inline evidence가 비용·정책·효과 균형. content-hash가 narration 파일을 포함해도 "편집 후 closeout_sig 재계산" 순서로 루프 없이 수렴.
**구현** — `turn_state_snapshot.py`(compute_signature/_audit_attested/collect_changed_paths/_file_hash 크기상한·fail-open/_settings_health), `scripts/closeout_sig.py`·`harness_doctor.py` 신규, `dead_code_scan.py` ruff, SKILL stamp v2, README setup, RISK 갱신.
**결과** — 라이브 실증: 내용-only 변경 sig flip, 무evidence stamp closeout_current=False·evidence stamp=True, commit 대칭 sig 일치, 이번 턴 stamp가 새 게이트를 **실제 통과**(green). 팀감사 4종이 REAL_BUG(sig 입력 비대칭) 1건 적발→`collect_changed_paths` 공유로 수정.
**한계** — evidence도 자기보고(가짜 채움 가능, 위조 *비용*만↑). enforce=block 미실증. settings.json 자동재현 불가(doctor 수동). vulture 미설치로 진짜 dead function 미탐.
**후속 과제** — subagent 산출물 hash를 게이트 증거로(03 §4 절충), settings.json tracked 전환 결정, vulture 설치+통폐합.
**관련 문서** — `.claude/hooks/turn_state_snapshot.py`, `.claude/skills/turn-closeout/SKILL.md`, `docs/_RISK/RISK_REGISTER.md`(R-CloseoutTrust/R-HarnessReproducibility/R-DeadCodeAudit).

### [#19] S2d CRUD 영속층: held_members 를 "degenerate held event + event_links(possible)" 로 영속(event↔event 스키마 vs record-key 의미 충돌 해소)
- 날짜: 2026-06-22 · 섹터: data-model/orchestration · 상태: **결정 + 구현(S2d 완료, S2e 통합 E2E 잔여)**

**문제** — S2c resolver 결정(APPEND/HOLD/CREATE)을 DB 에 적용하는 영속층에서, resolver 가 보류 대상으로 반환하는 `held_members` 는 **cross_source_dedup 의 record key(비-event 식별자)**인데, 영속 대상 `event_links` 는 `event_id`/`linked_event_id` **양단이 NOT NULL FK→events.id**(event↔event)다. record key 를 event_links 에 바로 넣을 수 없다 — 스키마와 의미가 충돌한다.
**배경** — S2-core 커밋(1487a2a) 후 S2d 착수. R-FalseMerge/R-EventTimelineS2Hardening 의 Closure 가 "held_members→event_links 영속 + is_snapshot_bidirectional 강제 + tz/UUID 방어"를 요구.
**가설** — held member 는 "primary event 와 같은 사건일 수 있으나 약신호/clique 미달로 자동병합하지 않은 후보"이므로, **그 자체를 하나의 (degenerate) Event 로 materialize 한 뒤 primary 와 possible 링크**로 묶으면 스키마를 지키면서 "자동병합 0 · 가역(status→rejected/merged) · 검수 표면"을 동시에 만족한다고 추정.
**도입 아이디어** — `event_timeline_service`(S2d): create_event/append_update(append-only)/get_event/set_snapshot/get_cluster_event/map_cluster(on_conflict_do_nothing=단일출처)/hold_link(possible)/apply_routing. **held_members 처리:** 각 member key 로 `canonical_title=member_key` 인 degenerate held event 를 create → `event_links(event_id=held, linked_event_id=primary, status='possible', reason='<decision.reason>:<key>')`. rich held payload(제목/시각/도메인)는 S2e 통합으로 이월(현재는 record key 만 보존).
**선택 이유** — 대안 (A) self-link(event_id==linked_event_id, key 를 reason 에)=의미 없는 자기참조 + 검수 불가. (B) event_links 미사용·별도 테이블=Closure 가 명시한 event_links 영속 위반 + 스키마 증식. (C) held 영속 자체 생략=R-FalseMerge Closure 미충족. degenerate held event 는 스키마 유효 + §17 "cluster_id 충돌→event_links" 패턴과 일관 + 가역.
**구현** — `backend/app/services/event_timeline_service.py`(CRUD + apply_routing + 방어 헬퍼 `_coerce_uuid`/`_ensure_aware`/`_sanitize_evidence`/`_sanitize_source_refs`). 방어: tz-naive→UTC, UUID/str 경계, evidence/source_refs 는 allowlist 키 + **scalar 값만** 영속(전문 본문·중첩 dict/list 은닉·임의 PII 필드 차단=R-EventTimelineS2Hardening ②③④). set_snapshot 은 카드가 다른 event 에 묶였으면 거부 + **세팅 후 실제 영속값 재조회로 is_snapshot_bidirectional 검증**(존재하지 않는 event 차단). append_update 는 event_updates INSERT 만(UPDATE/DELETE 0) + events.last_update_at GREATEST·first_seen LEAST(FSD pull-earlier). heat 재산정은 S2.5 이월(미터치). **apply_routing 은 내부 CRUD 를 commit=False 로 호출하고 1회 commit = 단일 원자 트랜잭션**(부분 실패 orphan 차단) + CREATE 전 cluster_event_map 조회로 재실행 orphan 회피. 회귀 27(append-only·실 쌍방향 검증·비-scalar 차단·apply_routing 분기/degrade/원자커밋·alembic 0001→0005 체인).
**결과** — 측정 게이트 backend(176)+ingestion(1328) **1504 green**(회귀 0; pre-existing embedding 환경실패 1 분리). resolver 결정이 실제 영속으로 연결: CREATE→events INSERT+cluster_event_map, APPEND→event_updates append(새 카드 0), HOLD→append 0+held 보류. 결정층(S2c) 무수정. **5-감사단**: architecture SOUND · evidence APPROVED · code/risk_closure(adversarial) 가 blocking 3종(set_snapshot 단언 trivially-true·apply_routing 다중 commit orphan·sanitize 非scalar 누수) 적발 → 즉시 교정·잠금.
**한계** — **통합 E2E 미배선(S2e)** — 실 cross_source_dedup 출력→resolver→apply_routing 의 end-to-end("2번째 보도→기존 Event append", transitive-only 자동승격 0)는 단위(mock 세션) 검증만(원자성·단조성의 실 DB 동작은 mock 으로 미커버). **잔여 동시성 race(정직):** apply_routing 단일 트랜잭션 + get-first 가드는 *순차 재실행* orphan 을 제거하나, **교차-트랜잭션 동시 CREATE**(둘 다 미매핑 조회 후 각자 create→on_conflict 로 한쪽만 매핑)는 DB unique 제약/advisory lock 이 필요 — S2e 로 이월. degenerate held event 는 record key 만 보유 → S2e 에서 rich payload 보강 + held event 중복(member 가 자기 resolution 으로 또 CREATE) 재평가. heat·merge_score entity/domain 축·실 Postgres migration 미실행.
**후속 과제** — S2e(FSD 단조 + 실 dedup→resolver→영속 통합 E2E + 교차-tx 동시 CREATE 멱등성 DB 제약 + held event 중복/승격 정책) · R-EventTimelineS2Hardening ①(event_updates/cluster_event_map/event_links CASCADE vs 감사 RESTRICT) 결정.

### [#20] Event 타임라인 삭제 정책: "app-layer hard-delete 미제공 + status 라이프사이클 + **DB 레벨 FK RESTRICT**" (감사 보호 app+DB 양층, live-PG 검증)
- 날짜: 2026-06-22 · 섹터: data-model/governance · 상태: **✅ 구현·live-PG 검증 완료** — app no-delete 정책 강제 + FK CASCADE→RESTRICT(alembic 0006) 실 Postgres up/down + 삭제 차단 동작 입증

**DONE(S2e live-PG, 2026-06-22):** 적대 B-2 가 지적한 "감사 보호 app 경로 한정·DB 레벨 미보호"를 **해소**. ① ORM 4 FK(event_updates.event_id·cluster_event_map.event_id·event_links.event_id/linked_event_id) `ondelete` CASCADE→**RESTRICT** + ② **alembic 0006**(drop/recreate FK, downgrade 가역) + ③ **live-PG 검증**: 0001~0006 실 Postgres(event_intel_test) up/down, confdeltype='r' 확인, `DELETE FROM events`(감사 이력 보유) 가 IntegrityError 로 **차단**됨(`test_live_fk_restrict_blocks_event_delete_with_history`). 이제 DB 직접/admin DELETE 도 의존 행(변화분·라우팅·링크)이 있으면 막힌다 — app+DB 양층 감사 보호.

**문제** — R-EventTimelineS2Hardening ① — `event_updates.event_id`(및 cluster_event_map/event_links) 의 ON DELETE **CASCADE** 가 "append-only 감사 로그" 의도와 긴장. Event 삭제 시 변화분·라우팅 메타가 통째 사라지면 상용 웹앱이 사용자에게 제공할 **사건 타임라인의 감사 가능성·재현성**이 깨진다.
**배경** — S2d 커밋(0b7c1d8) 후 S2e 통합 착수. 사용자가 "삭제보다 status 전환/soft-delete/rejected·merged 우선" 원칙 제시. Event timeline 은 evidence/update 이력 보존이 신뢰의 핵심.
**가설** — FK ondelete 를 바꾸기 전에, **그 위의 application 정책**(삭제 경로 자체를 제공하지 않음)을 확정하면 *app 경로* 에서는 CASCADE/RESTRICT 차이가 dormant 가 되고, 라이프사이클을 status 로만 운용하면 **앱이 제공하는 경로 한정**으로 감사/재현성을 코드로 보장할 수 있다고 추정. (DB 직접/admin 경로는 별도 — 아래 한계.)
**도입 아이디어** — ① **app-layer hard-delete 미제공:** `event_timeline_service`(및 향후 서비스)는 `delete_event`/`delete_update` 류 메서드를 두지 않는다(현재 0 — 강제 유지). 라이프사이클 = `events.status`(active→dormant→closed) + `event_links.status`(possible→confirmed/rejected/merged) 전환(가역). ② **event_updates 는 append-only**(INSERT 만; UPDATE/DELETE 0 — S2d 강제). ③ **CASCADE 는 DB 정합 backstop 으로 유지:** 만약 admin/migration 으로 event 가 *예외적으로* 삭제되면 dangling row 보다 cascade 가 정합적. 단 그 삭제 경로는 status 게이트 없이는 열지 않는다.
**선택 이유** — 대안 (A) 지금 FK 를 RESTRICT 로 바꾸는 migration 0006: live-PG 미검증(테스트가 mock 기반·DB fixture 부재)이라 "migration 동작"을 입증 못 함 + cluster_event_map/event_links 의 정당한 라이프사이클(merge/close)을 RESTRICT 가 방해할 수 있어 per-table 재설계 필요(S3+ 거버넌스 의존). (B) 결정 회피: ① 긴장이 계속 open. **app no-delete 정책은 *지금* 결정 가능·강제 가능**(삭제 메서드 부재가 1차 방어)하고 migration 은 live-PG 검증이 가능한 시점에 별도로(0006) — 점진·정직.
**구현** — 코드 변경 0(이미 삭제 메서드 부재). 본 ADR 가 정책을 명문화 + R-EventTimelineS2Hardening ① 를 "미결정"→"app no-delete 정책 결정(강제), FK RESTRICT migration 이월"로 갱신. 향후 삭제 경로 추가 시 본 ADR 의 status-gate 를 인수조건으로 강제.
**결과** — R-EventTimelineS2Hardening ① **DB 레벨까지 종결**: app no-delete(삭제 메서드 0) + FK RESTRICT(0006, live-PG 검증)로 **app+DB 양층** 감사 보호 달성. 감사 이력(event_updates)·라우팅(cluster_event_map)·링크(event_links)가 있는 Event 는 app 경로·DB 직접 양쪽에서 삭제 차단. 라이프사이클은 여전히 status 전환(active→dormant→closed / possible→merged)이 정도(正道).
**한계** — RESTRICT 는 "의존 행이 있으면 삭제 불가"라, **Event 병합/폐기(merge/close) 라이프사이클**에서 정당한 정리가 필요할 때 먼저 의존 행을 status 전환/이관해야 한다(S3+ 거버넌스에서 merge 흐름 설계 시 명시적 처리 — 현재는 삭제 경로 자체가 없어 무영향). live-PG 검증은 **disposable test DB(event_intel_test)** 기준 — 운영 DB 마이그레이션 실행은 배포 단계.
**후속 과제** — Event merge/close 라이프사이클(S3+)에서 RESTRICT 하 의존 행 정리 절차 + 운영 DB 0006 배포.

### [#22] Event live wiring: 수집 후보→Event 영속 경로 + 주입형 sink seam + event_cards 병행(자동연결 이월)
- 날짜: 2026-06-22 · 섹터: orchestration/data-model · 상태: **구현(C — 경로+seam, CLI 결선·자동연결 이월)**

**문제** — 라이브 수집 흐름이 `event_cards` 를 직접 생성(agent graph→upsert_card)하고, `cross_source_dedup`·`event_resolution_pipeline` 의 production 호출이 0이었다. 같은 사건이 N개 소스로 들어와도 하나의 Event 타임라인(events/event_updates)으로 누적되지 않았다 — 상용 인텔리전스의 핵심(중복 차단·타임라인 누적·증거 누적)이 미배선.
**배경** — S1~S2e 가 실 Postgres 까지 검증(33f019c)됐으나 "실 수집 데이터가 Event 로 흐르는 결선"은 미완. 사용자가 C(live wiring) 지시: 기존 event_cards 비파괴 + feature flag 분리 + 후보 단위 격리 + 본문/PII 차단.
**가설** — ingestion↔backend 가 이미 HTTP/주입형 콜러블(`db_writer`)로 결합돼 있으니, Event 영속도 **순수 결정/영속 계층(resolver/timeline_service/pipeline)의 ingestion-비의존을 보존**하면서 **단 하나의 backend 합성층**이 양쪽(ingestion.cross_source_dedup + backend.pipeline)을 잇고, orchestration 에는 `db_writer` 와 동일한 **주입형 sink** seam 만 추가하면 최소·안전하게 결선된다고 추정.
**도입 아이디어** — ① `backend/app/services/event_ingest_pipeline.py`(신규 합성층): `candidate_from_cluster` 매퍼(클러스터 primary record→ResolvedCandidate; title 상한 512·provenance delta·allowlist evidence·짧은 source_refs = 본문/PII 차단) + `build_record_index`(cross_source_dedup 와 **동일한** compute_record_key 로 member key 역매핑) + `ingest_records_to_events`(flag 게이트 + **후보 단위 try/except 격리** + `singletons_dropped` 가시화) + `make_orchestration_event_sink`(sync 어댑터). ② `EVENT_RESOLUTION_ENABLED`(config, 기본 false). ③ `run_production_orchestration(event_resolution_sink=None)` 주입 seam(cross_source_dedup 직후 호출, try/except 격리, result["event_resolution"]).
**선택 이유** — 대안 (A) agent graph(`agent_worker`)에 결선: 카드 1건씩 들어와 cross_source_dedup(단일멤버 제외)을 못 씀 + LLM 경로와 결합. (B) backend가 cross_source_dedup 미import(완전 duck-type): 매퍼의 member key 가 compute_record_key 와 어긋날 위험을 키움. (C) 선택안 — orchestration 이 이미 cluster_records 를 호출하고 db_writer 주입 선례가 있어 **seam 추가가 byte-identical 비파괴**. 순수 계층 비의존 보존 + 합성 경계 1곳 격리.
**구현** — `event_ingest_pipeline.py`(매퍼+ingest+sink) + config flag + orchestration seam + 테스트(backend 단위 18: 매퍼/flag OFF·ON/CREATE·APPEND·HOLD/격리/sanitize/singletons/cluster_id 순서불변/sink ON·OFF 경로 · orchestration 3: sink 주입·기본None·실패격리 · live-PG 3: candidate→CREATE→APPEND·flag off 영속0·실DB 후보격리). **event_cards 관계:** Event 경로는 events/event_updates 에만 write(event_cards 무변경·병행). `event_cards.event_id` **자동 연결은 이월**(set_snapshot 명시 연결만) — 카드↔Event 매칭은 별도 결정(R-EventModelMigration).
**결과** — production 호출 0 → **경로 구현 + 주입 seam 열림**. live-PG 로 "수집 후보 records → cross_source_dedup → resolver → events/event_updates" CREATE→APPEND(2번째 배치 새 Event 0)·flag off 영속 0·후보 단위 격리(실 DB rollback)가 실 Postgres 입증. 측정 게이트 backend+ingestion green. event_cards 경로·frontend 무영향(flag off 기본).
**한계(정직)** — ① **CLI `main()` 미결선:** orchestration seam 은 검증됐으나 운영 자동 실행은 아직 sink 를 주입하지 않는다 — session/engine 생명주기·flag 결선은 배포 단계(D/operational) 이월. 따라서 **production 자동 경로는 현재 Event 영속 0**(seam 만 열림). ② **event_cards.event_id 자동연결 이월:** Event 와 card 가 당분간 분리 운영(R-EventModelMigration). ③ **약신호 cluster_id 순서 안정성 미보장:** 강신호(canonical_url)는 입력 순서 불변(테스트 입증) 이나 약신호(meta: 키) cluster_id 는 `members[0]` 최소인덱스 의존 → 배치 간 멤버집합 변동 시 같은 사건 재-CREATE(split) 가능(R-FalseMerge 잔여). 약신호 Event 활성화 전 순서불변 회귀 필수. ④ **단일소스 사건 미생성:** cross_source_dedup 단일멤버 제외로 단독 보도는 Event 미생성(`singletons_dropped` 로 가시화만). ⑤ asyncio.run 어댑터는 sync 경계 전용(async 호출처는 async-native sink 필요). ⑥ backend→ingestion import(`cross_source_dedup`/`_external_url`/`compute_record_key`)는 wiring 모듈 1곳 한정(순수 계층 비의존 보존, 순환 없음); private 함수 import 는 향후 public 승격 권고.
**후속 과제** — D: CLI/runner 에 backend-bound sink 결선(session 생명주기) + frontend Event timeline + 주기 수집 데모. + event_cards↔Event 자동연결 정책(ADR) + 약신호 cluster_id 안정 키 + 단일소스 사건 정책.
**관련 문서** — `backend/app/services/event_ingest_pipeline.py`, `ingestion/tools/run_production_orchestration.py`, `backend/tests/test_event_ingest_pipeline.py`, `backend/tests/test_event_resolution_live_pg.py`, ADR#16/#19/#21, `docs/_RISK/RISK_REGISTER.md`(R-FalseMerge·R-EventModelMigration·R-EventTimelineS2Hardening·R-ExpansionPartialFailure), `docs/2_ROADMAP/19`.

### [#23] D-1 운영 결선 composition root: backend-side sink 주입(decoupling 불변식 보존)
- 날짜: 2026-06-22 · 섹터: orchestration/operational · 상태: **구현(D-1 — composition root + 능력, 주기 auto-trigger 이월)**

**문제** — ADR#22 가 Event 영속 경로 + 주입형 sink seam 을 열었으나 **운영 자동 실행(`run_production_orchestration.main()`)이 sink 를 주입하지 않아 production 자동 경로 Event 영속이 0**이었다. seam 만 열려 있고 결선이 비어 있었다.
**배경** — 사용자가 D-1(운영 결선) 지시. docs 권위(00_ROADMAP_INDEX §4 "CLI 결선"·15 Phase S1 "CLI/runner sink 결선 이월"·PROJECT_STATUS 다음할일#1)가 일치해 D-1 을 S2 즉시 잔여로 가리킴(불일치 없음).
**불일치(구현 전 보고)** — PROJECT_STATUS 문자 plan + sink docstring 은 sink 를 `ingestion/tools/run_production_orchestration.main()` 에 `--event-resolution` 으로 결선하라 적었으나, 이는 **decoupling 불변식**(ingestion 런타임은 backend 를 import 하지 않음 — ingestion 런타임/라이브러리 backend import 0건 검증)과 충돌(사상 첫 ingestion→backend in-process import = 의존 방향 역전). 기존 `--raw-events-sink backend` 선례는 backend 를 import 하지 않고 `ingestion.integration` HTTP 클라이언트를 쓸 뿐이라 선례 아님. → 사용자에게 composition root 위치 확인.
**결정(사용자 선택)** — **backend-side composition root.** ① ingestion `main(argv, *, event_resolution_sink=None)` 으로 seam 을 main 레벨까지 연장(주입형 **Callable** 만 받음 — ingestion→backend import 0, decoupling 보존). ② 신규 `backend/app/tools/run_event_orchestration.py`: backend→ingestion(허용 방향)으로 `run_production_orchestration.main` 을 import 해 재사용하고, **backend 가 sink 를 만들어 주입**한다. **전용 NullPool async engine + async_sessionmaker** 를 backend 가 소유(생명주기) — API 서버용 module-global 풀 엔진을 재사용하지 않음(sink 가 호출당 asyncio.run 으로 구동되므로 NullPool 이 loop 간 커넥션 누수/재사용을 차단). flag off(`EVENT_RESOLUTION_ENABLED` 기본 false + `--event-resolution` 미지정)면 sink=None → **orchestration core 동작 byte-identical**(DB 미접근, event_cards 경로 보존). backend CLI 래퍼만 event_resolution 상태 1줄을 추가 출력(데이터/DB 무변경).
**선택 이유** — 대안 (B) ingestion CLI lazy 조건부 import: `--raw-events-sink backend` 와 동형 opt-in 이나 **첫 ingestion(tools)→backend import** 도입(가드돼도 의존 역전). (C) HTTP 내부 엔드포인트: 가장 엄격한 분리(raw_events 동형)이나 신규 endpoint+auth+직렬화로 batch 도구엔 과설계. **(A) 선택** — backend→ingestion 은 이미 허용된 단방향(브리지 모듈 선례)이고, ingestion/tools 무변경으로 backend-free 유지 + 엔진 생명주기를 DB 소유자(backend)에 둠.
**구현** — ingestion `main()` seam 연장 + event_resolution 결과 출력(운영 관찰성) · `backend/app/tools/run_event_orchestration.py`(`event_resolution_sink_cm` 생명주기 CM + `run_event_orchestration` + `main` --event-resolution 소비/위임) · 테스트(backend 단위 9: flag OFF/ON 주입·engine dispose·예외에도 dispose·CM 게이트·main 플래그 소비/위임·off-by-default · live-PG 1: 실 sink 를 테스트 DB factory 로 구동→Event CREATE→APPEND). ingestion 회귀 +0(1331 green).
**결과** — production 자동 경로가 **Event 영속 능력 확보**: 운영 runner 를 `--event-resolution`(또는 `EVENT_RESOLUTION_ENABLED=true`)로 실행하면 수집 후보가 cross_source_dedup→resolver→events/event_updates 로 실제 영속(live-PG 로 sink 의 실 DB CREATE→APPEND 입증). flag off 기본은 DB/데이터 동작 보존(event_cards 경로·frontend 무영향; backend CLI 만 상태 1줄 출력).
**한계(정직)** — ① **주기 auto-trigger 미배선:** D-1 은 runner 의 결선 *능력* 을 만든다 — 그러나 runner 를 주기적으로 자동 실행하는 스케줄러(Celery beat/cron)는 아직 없다(Phase 2 잔여). "주기적으로 Event 가 쌓인다"는 (D-1 능력) + (주기 트리거) + (실 수집 production-validation 실행) + (운영 DB alembic 적용)이 모두 필요. ② **실 수집 production-validation 미실행:** live-PG 검증은 synthetic records(테스트 DB) 기준 — 실 소스 probe→Event 누적은 운영 배포 시 입증(이월). ③ event_cards↔Event 자동연결 이월(R-EventModelMigration) · 약신호 cluster_id 안정키 이월(R-FalseMerge). ④ asyncio.run 어댑터 sync 경계 전용은 ADR#22 와 동일.
**후속 과제** — D-2: frontend Event timeline + `/api/events` Event 경유 + Docker 데모. + Celery beat/cron 주기 트리거(Phase 2) + 실 production-validation 1회 Event 누적 입증 + event_cards↔Event 자동연결 정책.
**관련 문서** — `backend/app/tools/run_event_orchestration.py`, `ingestion/tools/run_production_orchestration.py`, `backend/tests/test_run_event_orchestration.py`, `backend/tests/test_event_resolution_live_pg.py`(test_live_d1_orchestration_sink_persists_event), ADR#22, `docs/_RISK/RISK_REGISTER.md`(R-EventTimelineS2Hardening), `docs/2_ROADMAP/{00,15,19}`.

### [#30] delta_summary deterministic 자연어화 — 실경로 디버그 라벨 제거(R-EventTimelineRenderHardening② 코드 done·실 렌더 잔여) + CREATE genesis update 이월
- 날짜: 2026-06-23 · 섹터: pipeline/web/ux · 상태: **구현(자연어화) + 설계 이월(genesis update)**

**문제** — 실 소스 검증(ADR#29)이 확인: `event_ingest_pipeline.candidate_from_cluster` 의 `delta_summary=f"{cluster.confidence}:{cluster.reason}"` 는 디버그 라벨(예 `"possible_duplicate:title_date_similarity"`)이라 실 데이터 타임라인 본문이 사용자용 자연어가 아니다(R-EventTimelineRenderHardening②). **가설** — LLM 없이 **deterministic template**(resolver 가 확인한 사실=confidence/멤버수/출처종류)으로 사용자용 자연어를 만들 수 있다. 원문 본문/허위/투자판단 미생성, 과장 표현 금지.
**구현** — `build_delta_summary(confidence, reason, member_count, record_type)` 신규(`event_ingest_pipeline.py`). 강신호(`duplicate`)=distinct≥2 `서로 다른 {종류} 출처 N곳이 동일 식별자로 같은 사건을 보도했습니다.` / distinct 1(동일 URL이 여러 피드) `{종류} 보도가 동일 식별자로 확인된 사건입니다.` / 약신호(`possible_duplicate`)=`유사한 제목·같은 시점의 {종류} 보도 N건이 같은 사건으로 추정됩니다(자동 병합 전 교차 검토).` / 미지 confidence=안전 fallback. record_type→사용자 라벨(뉴스/공식/구조화 지표/검색/커뮤니티). **member_count=`len(distinct_members)`**(distinct 근거 수=evidence 링크 수와 일치 — "N곳" 과대계수 차단, adversarial P2-1). **LLM/network 0.**
**검증** — 단위(강신호 distinct≥2/distinct 1·약신호·출처종류별·미지 fallback·원시 enum 누출 미탐(`_is_debug_label`)·과장표현 금지) + 통합(약신호 멀티소스 매퍼=실 검증 코스피/대우건설 동형) + **live-PG 결정론 확인**: 강신호(동일 URL) CREATE→APPEND → event_intel_test 실 update 자연어(디버그 라벨 아님). 변경 표면 green. 디버그 라벨 단언 테스트를 자연어 단언으로 갱신.
**결과** — **R-EventTimelineRenderHardening 부분종결**: ①③(ADR#26, 실데이터 비의존) 종결 + ② **코드 자연어화 done**. 단 **② 실 렌더 도달은 잔여**(adversarial P1-1): delta_summary 는 `EventUpdateItem`(update 행)에서만 렌더되는데 실 Event(ADR#29)는 전량 CREATE-only(update 0)라 화면 미표시 — 실 fetch APPEND/genesis 전까지 "RENDER hardening" 효과 미도달. R-RealSourceLoopUnproven ⑦(delta_summary 코드 가독) 해소, 가시화는 잔여.
**CREATE genesis update 이월(설계 판정)** — ADR#29 발견: 실 Event 2건은 CREATE-only 라 update 0 → 자연어 delta_summary 가 화면에 안 보임(APPEND 에서만 영속). **CREATE 시 genesis update 를 남기면 가시화되나**, 그것은 `apply_routing` 의 핵심 불변식 **"CREATE 는 update 0"**(ADR#16/#19, live-PG `test_live_create_persists_no_update` 등 다수 단언) + 시맨틱(생성≠변경)을 바꾸는 **큰 설계 변경**이다 → 사용자 지침("설계 영향 크면 이월")에 따라 **이번 턴 미구현, R-RealSourceLoopUnproven closure 조건으로 이월**. (대안: genesis evidence 를 Event 자체에 두는 스키마 변경도 검토 대상 — 별도.)
**한계(정직)** — delta_summary 는 "교차 출처 관찰"을 설명(사건 *내용* 은 canonical_title 이 보유; 전문 저장 금지 원칙). CREATE-only Event 는 여전히 update/evidence 미표시(genesis 이월). live-PG 확인은 synthetic 강신호 record(실 fetch 아님 — 실 fetch APPEND 은 ADR#29 잔여).

### [#31] CREATE genesis update — "CREATE 는 update 0" 불변식 의도적 개정(실 Event 빈 상세 해소, R-EventTimelineRenderHardening② 종결)

### [#33] source-type publish gate — pure community/search/structured 단독 cross-source 직접 발행 차단(ACTION_WITHHELD); primary-authority 이월
- 날짜: 2026-06-23 · 섹터: pipeline / source-type fidelity / 발행 안전 / 투자조언 경계 · 상태: **구현 완료(gate); primary-authority 이월**
