**구현** — 훅에 `_is_narration_output` 제외 + sig 비교 freshness. 스킬은 `narrated_sig` 기록.
**결과** — marker에 sig 기록 후 `narrative_fresh=True`·nudge 소멸 검증. 정적 dirty tree 침묵.
**한계** — 기존 untracked 파일의 내용-only 변경(경로 불변)은 미포착(과소 nudge).
**후속 과제** — 내용-only 변경 과소 nudge 운영 관찰.

### [#7] closeout 게이트를 "경로 sig + 자기보고" → "content-hash sig + 구조화 evidence"로 격상(증거기반)
- 날짜: 2026-06-19 · 섹터: harness · 상태: 완료

**문제** — 직전 stamp 게이트는 (a) sig가 경로집합 기반이라 **동일 경로의 내용-only 변경**(claim/risk 근거/config)을 못 잡고, (b) `audit_types_addressed`에 유형명만 적으면 통과해 **감사를 실제로 안 해도 거짓 stamp가 통과**했다.
**배경** — 운영 백본이 "증거 기반 stamp-gated"가 되려면, 무엇이 바뀌었는지(내용)와 무엇을 검토했는지(증거)를 게이트가 기계적으로 확인해야 한다. 직전 감사가 R-CloseoutTrust로 이 둘을 잔여 gap으로 남겼었다.
**가설** — sig에 중요 파일 content-hash를 합성하면 내용변경이 잡히고, required 감사마다 구조화 evidence(executed+verdict)를 요구하면 자기보고 위조 비용이 오른다. 단 "LLM이 실제로 추론했는가"는 원천적으로 검증 불가.
**도입 아이디어** — ① `compute_signature`(경로 sig + `content:<path>#hash` of 중요 파일군, narration 포함) + 에이전트가 편집 후 `scripts/closeout_sig.py`로 재계산해 stamp에 복사(hook과 동일 함수→수렴). ② `_audit_attested` 게이트(required audit별 evidence 필수). ③ `scripts/harness_doctor.py`로 gitignored settings.json 재현성 점검. ④ dead_code_scan에 ruff 연동.
**선택 이유** — 대안 A(전체 repo hash)=과중·narration 루프, 대안 B(별도 감사 리포트 파일 요구)=`03 §4`(파일 증식 금지)와 상충. 중요 파일군만 hash + inline evidence가 비용·정책·효과 균형. content-hash가 narration 파일을 포함해도 "편집 후 closeout_sig 재계산" 순서로 루프 없이 수렴.
**구현** — `turn_state_snapshot.py`(compute_signature/_audit_attested/collect_changed_paths/_file_hash 크기상한·fail-open/_settings_health), `scripts/closeout_sig.py`·`harness_doctor.py` 신규, `dead_code_scan.py` ruff, SKILL stamp v2, README setup, RISK 갱신.
**결과** — 라이브 실증: 내용-only 변경 sig flip, 무evidence stamp closeout_current=False·evidence stamp=True, commit 대칭 sig 일치, 이번 턴 stamp가 새 게이트를 **실제 통과**(green). 팀감사 4종이 REAL_BUG(sig 입력 비대칭) 1건 적발→`collect_changed_paths` 공유로 수정.
**한계** — evidence도 자기보고(가짜 채움 가능, 위조 *비용*만↑). enforce=block 미실증. settings.json 자동재현 불가(doctor 수동). vulture 미설치로 진짜 dead function 미탐.
**후속 과제** — subagent 산출물 hash를 게이트 증거로(03 §4 절충), settings.json tracked 전환 결정, vulture 설치+통폐합.
**관련 문서** — `.claude/hooks/turn_state_snapshot.py`, `.claude/skills/turn-closeout/SKILL.md`, `docs/_RISK/RISK_REGISTER.md`(R-CloseoutTrust/R-HarnessReproducibility/R-DeadCodeAudit).

### [#18] S2 Event Resolution을 "S2-core(라우팅+clique+CRUD) 먼저, merge_score는 signal_strength+clique만"으로 분할 + clique=강신호 단일 연결성분 정의

**문제** — S2(Event Resolution)는 라우팅+clique 게이트+merge_score 3축+heat 4신호+CRUD로 한 번에 가기엔 큰 덩어리고, merge_score의 2축(entity_overlap=Entity Registry S4 의존, domain_distance=거버넌스 ADR θ UNKNOWN)은 선행 미충족. 또 R-FalseMerge clique 게이트의 "완전연결" 정의가 구현상 모호.
**배경** — S1 토대 커밋(87807ea) 후 S2 착수. 사용자 승인: S2-core 먼저, merge_score는 signal_strength+clique만(2축 이월).
**가설** — 라우팅 정확성(R-FalseMerge 차단)이 heat/merge_score 정교화와 독립적이므로, decision layer(dedup clique 확장 + resolver 라우팅)를 먼저 닫으면 위험한 오병합을 heat 이전에 차단할 수 있다고 추정.
**도입 아이디어** — S2a alembic 0005(cluster_event_map/event_links) + ORM/Pydantic · S2b cross_source_dedup 확장(signal_strength=Jaccard 연속값 보존, clique_ok, weak_only_members; additive) · S2c event_resolver 순수 라우팅(APPEND/HOLD/CREATE + clique 게이트, ingestion 비의존). entity_overlap/domain_distance/heat는 구조/이월.
**선택 이유** — 대안 (S2-full 한 번에)=검증 표면 과대 + 미충족 2축 강행. (S2a만)=라우팅 위험(R-FalseMerge) 미차단. S2-core가 위험 우선 차단 + 점진. **clique 정의:** 적대 감사 B1이 "강신호 끝점인가"(naive anchor)는 **두 강성분이 약신호로만 브릿지된 경우**(A-B강, C-D강, B-C약)를 오통과함을 입증 → **강신호 edge만의 단일 연결성분이 전체 멤버를 덮는가**로 교정(완전 K-clique 강제는 정당한 강신호 체인 과판정이라 회피).
**구현** — `backend/app/models/event_resolution.py`(ClusterEventMapORM/EventLinkORM), `alembic 0005`, `backend/app/services/event_resolver.py`, `ingestion/orchestration/cross_source_dedup.py`(clique 확장+B1 교정), Pydantic(ClusterEventMap/EventLink). 회귀: dedup 12(transitive/두-강성분 브릿지 차단·연속값 보존)·resolver 9(라우팅 분기·N3 값계약)·resolution 11.
**결과** — 측정 게이트 backend+ingestion **1477 green**(회귀 0; pre-existing embedding 환경실패 1 분리). 4-감사단: architecture SOUND·pipeline SOUND·evidence APPROVED·code(adversarial) B1/B2/N3 수정·잠금.
**한계** — decision layer만 — held_members→event_links 영속(S2d) + resolver가 실 dedup 출력 받는 통합 E2E 미배선(현재 양쪽 단위 테스트만). signal_strength 단일 대표값·max 편향은 게이팅 사용 시 재평가(R-FalseMerge remaining). 실 Postgres migration 미실행.
**후속 과제** — S2d(event_timeline_service CRUD: held_members→event_links possible 적재 + is_snapshot_bidirectional 강제 + tz/UUID 방어=R-EventTimelineS2Hardening ②③) · S2e(FSD + 통합 E2E "2번째 보도→append").
**관련 문서** — `docs/2_ROADMAP/{12,19}`, `docs/5_REFERENCE/EVENT_SCHEMA.md` Part 2, ADR#16/#17, `docs/_RISK/RISK_REGISTER.md`(R-FalseMerge·R-EventTimelineS2Hardening).

### [#19] S2d CRUD 영속층: held_members 를 "degenerate held event + event_links(possible)" 로 영속(event↔event 스키마 vs record-key 의미 충돌 해소)
- 날짜: 2026-06-22 · 섹터: data-model/orchestration · 상태: **결정 + 구현(S2d 완료, S2e 통합 E2E 잔여)**

**문제** — S2c resolver 결정(APPEND/HOLD/CREATE)을 DB 에 적용하는 영속층에서, resolver 가 보류 대상으로 반환하는 `held_members` 는 **cross_source_dedup 의 record key(비-event 식별자)**인데, 영속 대상 `event_links` 는 `event_id`/`linked_event_id` **양단이 NOT NULL FK→events.id**(event↔event)다. record key 를 event_links 에 바로 넣을 수 없다 — 스키마와 의미가 충돌한다.
**배경** — S2-core 커밋(1487a2a) 후 S2d 착수. R-FalseMerge/R-EventTimelineS2Hardening 의 Closure 가 "held_members→event_links 영속 + is_snapshot_bidirectional 강제 + tz/UUID 방어"를 요구.
**가설** — held member 는 "primary event 와 같은 사건일 수 있으나 약신호/clique 미달로 자동병합하지 않은 후보"이므로, **그 자체를 하나의 (degenerate) Event 로 materialize 한 뒤 primary 와 possible 링크**로 묶으면 스키마를 지키면서 "자동병합 0 · 가역(status→rejected/merged) · 검수 표면"을 동시에 만족한다고 추정.
**도입 아이디어** — `event_timeline_service`(S2d): create_event/append_update(append-only)/get_event/set_snapshot/get_cluster_event/map_cluster(on_conflict_do_nothing=단일출처)/hold_link(possible)/apply_routing. **held_members 처리:** 각 member key 로 `canonical_title=member_key` 인 degenerate held event 를 create → `event_links(event_id=held, linked_event_id=primary, status='possible', reason='<decision.reason>:<key>')`. rich held payload(제목/시각/도메인)는 S2e 통합으로 이월(현재는 record key 만 보존).
**선택 이유** — 대안 (A) self-link(event_id==linked_event_id, key 를 reason 에)=의미 없는 자기참조 + 검수 불가. (B) event_links 미사용·별도 테이블=Closure 가 명시한 event_links 영속 위반 + 스키마 증식. (C) held 영속 자체 생략=R-FalseMerge Closure 미충족. degenerate held event 는 스키마 유효 + §17 "cluster_id 충돌→event_links" 패턴과 일관 + 가역.
**구현** — `backend/app/services/event_timeline_service.py`(CRUD + apply_routing + 방어 헬퍼 `_coerce_uuid`/`_ensure_aware`/`_sanitize_evidence`/`_sanitize_source_refs`). 방어: tz-naive→UTC, UUID/str 경계, evidence/source_refs 는 allowlist 키 + **scalar 값만** 영속(전문 본문·중첩 dict/list 은닉·임의 PII 필드 차단=R-EventTimelineS2Hardening ②③④). set_snapshot 은 카드가 다른 event 에 묶였으면 거부 + **세팅 후 실제 영속값 재조회로 is_snapshot_bidirectional 검증**(존재하지 않는 event 차단). append_update 는 event_updates INSERT 만(UPDATE/DELETE 0) + events.last_update_at GREATEST·first_seen LEAST(FSD pull-earlier). heat 재산정은 S2.5 이월(미터치). **apply_routing 은 내부 CRUD 를 commit=False 로 호출하고 1회 commit = 단일 원자 트랜잭션**(부분 실패 orphan 차단) + CREATE 전 cluster_event_map 조회로 재실행 orphan 회피. 회귀 27(append-only·실 쌍방향 검증·비-scalar 차단·apply_routing 분기/degrade/원자커밋·alembic 0001→0005 체인).
**결과** — 측정 게이트 backend(176)+ingestion(1328) **1504 green**(회귀 0; pre-existing embedding 환경실패 1 분리). resolver 결정이 실제 영속으로 연결: CREATE→events INSERT+cluster_event_map, APPEND→event_updates append(새 카드 0), HOLD→append 0+held 보류. 결정층(S2c) 무수정. **5-감사단**: architecture SOUND · evidence APPROVED · code/risk_closure(adversarial) 가 blocking 3종(set_snapshot 단언 trivially-true·apply_routing 다중 commit orphan·sanitize 非scalar 누수) 적발 → 즉시 교정·잠금.
**한계** — **통합 E2E 미배선(S2e)** — 실 cross_source_dedup 출력→resolver→apply_routing 의 end-to-end("2번째 보도→기존 Event append", transitive-only 자동승격 0)는 단위(mock 세션) 검증만(원자성·단조성의 실 DB 동작은 mock 으로 미커버). **잔여 동시성 race(정직):** apply_routing 단일 트랜잭션 + get-first 가드는 *순차 재실행* orphan 을 제거하나, **교차-트랜잭션 동시 CREATE**(둘 다 미매핑 조회 후 각자 create→on_conflict 로 한쪽만 매핑)는 DB unique 제약/advisory lock 이 필요 — S2e 로 이월. degenerate held event 는 record key 만 보유 → S2e 에서 rich payload 보강 + held event 중복(member 가 자기 resolution 으로 또 CREATE) 재평가. heat·merge_score entity/domain 축·실 Postgres migration 미실행.
**후속 과제** — S2e(FSD 단조 + 실 dedup→resolver→영속 통합 E2E + 교차-tx 동시 CREATE 멱등성 DB 제약 + held event 중복/승격 정책) · R-EventTimelineS2Hardening ①(event_updates/cluster_event_map/event_links CASCADE vs 감사 RESTRICT) 결정.
**관련 문서** — `docs/2_ROADMAP/19`(§2.2·§21.1), `docs/5_REFERENCE/EVENT_SCHEMA.md` Part 2(§cluster_event_map·event_links), ADR#16/#18, `docs/_RISK/RISK_REGISTER.md`(R-FalseMerge·R-EventTimelineS2Hardening).
