## A. 2a — 코드기반 docs lifecycle

### A.1 핵심 원칙
- **단방향:** 코드/산출물이 진실 → docs 메모리가 따라간다. docs 가 코드를 바꾸지 않는다.
- **메모리 파일 정의:** 구현상세/plan/탐색 아이디어 docs/**.md. (계약 스펙 `API_CONTRACT`·`EVENT_SCHEMA` 등 불변 문서는 대상 아님.)
- **삭제 금지:** `rm`/`Remove-Item` deny + 금지. **진짜 삭제 안 함, `Move-Item` 되돌림 가능 이동만.**

### A.2 단일 출처 통일 (R4 — 구현 전 선결)
> **문제:** 기존 `docs-sync-skill`·`docs-memory-curator` 는 단일출처를 `*_FINAL.md` 로 정의하나, `_CANONICAL/00 §2` 는 권위 정점을 `_CANONICAL/*` 로 정의 → **이미 어긋나 있음.** `turn-closeout` 이 3번째 주체로 들어오기 전에 통일한다.
- **권위 순서(통일안):** `① 코드+최신 산출물 → ② _CANONICAL/* → ③ 영역 FINAL(*_FINAL.md, INGESTION_FINAL 등) → ④ 기타 설계문서`. (= `_CANONICAL/00 §2` 와 일치.)
- **조치:** `docs-sync-skill/SKILL.md` 와 `docs-memory-curator.md` 의 "단일 출처 = `*_FINAL.md`" 문구를 "**권위 정점 = `_CANONICAL/*`**, `*_FINAL.md` 는 영역 상세(순서 ③)" 로 정정(`06` 마이그레이션 단계).

### A.3 세 주체의 동사 분리 (R4 — 중첩 해소)
| 주체 | 동사 | 권한 | 무엇 |
|---|---|---|---|
| `docs-sync-skill` | **교정** | Read/Edit | status 라벨 drift in-place 수정(google_trends PASS 오표기 등) |
| `turn-closeout` 스킬 | **이동** | Bash(`Move-Item`/`git mv`) | lifecycle 파일 단위 이동(ACTIVE→ARCHIVE→TRASH) |
| `docs-memory-curator` 에이전트 | **판정** | Read/Grep(이동 안 함) | 이동 후보의 코드 착지 근거 검증·자문 |
> 계약: curator 가 "이동 권고" 를 출력 → turn-closeout 이 `Move-Item` 으로 집행. 셋이 "동기화" 를 공유하되 동사가 다르다.

### A.4 lifecycle (3단계, 매니페스트 샤딩 없음)
```
ACTIVE (docs/**)
   │  매 턴: 훅이 후보 신호, 스킬이 grep 근거로 판정, curator 자문
   ▼ (구현이 코드에 착지 = 문서 역할 종료, 또는 dead/미사용)
ARCHIVED  →  Move-Item docs/X.md  docs/_ARCHIVE_SUPERSEDED/X.md  + _INDEX.md 1줄
   ▼ (보존기간 경과 = config.archive_retention_turns 기본 30)
TRASHED   →  Move-Item ... docs/_TRASH/  (gitignored)
```
- **판정 근거 필수:** "이미 적용"=문서가 기술한 구현이 코드에 존재(grep 심볼/경로)+테스트 통과. "미사용"=문서가 가리키는 경로/심볼이 코드에 없음. **추측 금지, grep 근거 첨부.** 경계 사례는 ARCHIVE 보류 + risk 등록.
- **안전장치:** archive/trash 이동은 **`03` 팀 감사(docs-memory-curator + adversarial-reality-critic) 통과 후에만.** 단독 이동 금지.

### A.5 tombstone (`docs/_ARCHIVE_SUPERSEDED/_INDEX.md`)
```
| 아카이브일(turn) | 원래 경로 | 사유(한 줄) | 코드 착지 근거(path/risk-id) | 보존만료(turn) |
```

### A.6 TRASH 복구 한계 (R9 — 명시)
- `docs/_TRASH/` 는 gitignored → **단일 로컬 머신에서만 복구 가능**. codex worktree·새 clone 에선 추적 안 됨(사실상 소멸).
- 따라서 "되돌림 가능" 주장은 **로컬 한정**. 영구 제거가 필요하면 사용자 승인 후 `git rm`(deny 아님) 으로 추적 이력에 남기는 경로를 별도 고려.

## B. 2b — 의사결정 지식 자산 (사용자 Q1 지시로 격상)

> **목표(사용자 지시 원문 요지):** 문서는 단순 변경 이력이 아니라 **프로젝트의 설계 의도·의사결정 맥락을 장기 보존하는 지식 자산**이다. "무엇을 바꿨나"보다 **"왜 이 선택을 했나"**를 우선한다. 세션이 바뀌거나 공백이 생겨도 **새 에이전트가 사고 흐름을 복원**할 수 있어야 한다. 폐기된 아이디어·시행착오도 핵심 이유를 남긴다. 특정 구현체/폴더 구조에 종속되지 않게 한다. 길이보다 **미래 작업자의 이해 가능성**이 평가 기준이다.

### B.1 위치·단위
- **파일:** `docs/_DECISIONS/SESSION_<날짜>_<짧은해시>.md` — 세션당 1파일(세션 격리).
- **append 단위:** "설계 판단/아이디어 착지/방향 전환/폐기" 가 일어난 **결정 단위**. 사소한 턴엔 추가 안 함(분량 폭증 방지).
- **구현체-비종속:** 블록은 "왜/대안/트레이드오프" 중심. 코드 경로는 **근거로만** 인용하고, 경로가 바뀌어도 사고 흐름이 읽히도록 작성(파일명 나열 금지).

### B.2 블록 템플릿 (사용자 지정 양식 — 정확히 이 필드, "왜" 우선, 간결)
```markdown
---
### [#N] <제목>
- 날짜: 2026-06-19 · 섹터: (harness/ingestion/…) · 상태: [진행중 / 완료 / 폐기 / 보류]

**문제** — 무엇이 문제였는가?
**배경** — 왜 이 문제가 중요한가? 프로젝트 전체 목표와 어떤 관련이 있는가?
**가설** — 원인을 어떻게 추정했는가?
**도입 아이디어** — 어떤 해결책을 선택했는가?
**선택 이유** — 다른 대안은 무엇이 있었는가? 왜 이 방식을 채택했는가?
**구현** — 실제 변경 사항.
**결과** — 무엇이 개선되었는가? 정량/정성 결과.
**한계** — 아직 남은 문제.
**후속 과제** — 다음 세션에서 이어서 해야 할 일.
**관련 문서** — 참고해야 할 md 파일.
```
> **평가 기준(사용자 지시):** 길이가 아니라 "미래 작업자가 이 결정을 이해·복원할 수 있는가". 각 필드 1~2줄, 블록 ~20줄 내외. **간략하게 — 너무 길어지면 안 됨.** "왜"가 빠지면 길이 무관하게 미달.

### B.3 lifecycle
- 결정 로그는 **자산(보존 우선)** — 함부로 이동 안 함. 상태 `완료/폐기` + 코드 착지 확인 시에만 A.4 archive 후보(사고 흐름 요약 1줄 + archive 링크만 남김).
- 폐기(`상태:폐기`) 블록도 **삭제하지 않는다** — "왜 버렸나" 가 재검토 자산(사용자 지시).
